# UniPapers Pre-Production Audit

## Executive Summary

This codebase is not ready for public deployment. The dominant risks are not subtle application bugs; they are basic production failures in secret management, deployment hardening, storage architecture, and abuse resistance. The application can probably function in a local demo, but the current design will break or degrade under real production conditions, especially if it is deployed on the Vercel and Cloudinary path implied by the settings.

Risk level: Critical. Exploitability: high for the secret and session issues, high for the deployment/storage failures, and moderate-to-high for abuse and denial-of-service paths. The most exposed components are [UniPapers/settings.py](UniPapers/settings.py#L25), [home/views.py](home/views.py#L360), [home/models.py](home/models.py#L65), [home/templates/home/search.html](home/templates/home/search.html#L678), [home/templates/home/profile.html](home/templates/home/profile.html#L817), and [home/templates/home/base.html](home/templates/home/base.html#L2).

I did not find obvious raw SQL execution, shell command injection, or direct command execution in the reviewed paths. That is the one positive finding. It does not meaningfully change the overall verdict because the real problems are operational and architectural: the application leaks secrets, runs with insecure defaults, stores files in ways that do not survive the stated deployment model, and lacks enough abuse controls to tolerate hostile traffic.

Priority order: fix the secret and session hardening first, then fix the storage/file-move architecture, then remove the abuse and performance bottlenecks, then address maintainability and test coverage.

## Critical Findings

- Critical: Hardcoded Django SECRET_KEY in [UniPapers/settings.py](UniPapers/settings.py#L25). What it is: the signing secret is committed directly in source. Why dangerous: anyone who gets the code can forge or tamper with signed Django data, including session cookies and password-reset related tokens. Exploitability: trivial once the repository or a deployment artifact is exposed; no runtime compromise is needed. Real-world impact: session forgery, trust collapse for signed data, and permanent compromise of authentication integrity. Worst case: full account takeover and a broadly trusted but untrustworthy application. Fix: move the key into an external secret manager or environment variable, rotate it immediately, and invalidate existing sessions and tokens after rotation. Type: implementation-level, but production-critical.

- Critical: Production storage flow is built on local filesystem moves even though the settings switch to Cloudinary on Vercel in [UniPapers/settings.py](UniPapers/settings.py#L137) and [home/models.py](home/models.py#L65). What it is: approval logic uses os.path, os.replace, and settings.MEDIA_ROOT to move files from pending to approved storage. Why dangerous: this design assumes a writable local filesystem and will break or misbehave when the app is deployed on remote storage. Exploitability: an attacker does not need to exploit this; normal moderation traffic can trigger it and crash the approval path or leave files orphaned. Real-world impact: failed approvals, broken previews, lost attachments, and a moderation workflow that cannot be trusted in production. Worst case: the public launch works until the first real approval batch, then file handling starts failing and data integrity becomes unpredictable. Fix: remove filesystem-specific move logic, use storage-agnostic rename/copy APIs where supported, or keep a single immutable storage path and update metadata instead of moving files. Type: architectural.

- Critical: Approved-file relocation uses the original basename without collision handling in [home/models.py](home/models.py#L88). What it is: the file is moved to uploads/papers/{filename} with os.replace, which overwrites an existing file with the same name. Why dangerous: common paper names are generic, so collisions are realistic. Exploitability: a user can upload a file with a chosen name and let the approval flow overwrite another record’s file. Real-world impact: silent corruption, content replacement, and cross-record data loss. Worst case: one submitted paper replaces another paper’s file, which is both an integrity failure and a potential privacy incident if records later point to the wrong content. Fix: generate collision-resistant storage names, never overwrite on move, and verify uniqueness before finalizing approval. Type: implementation-level with architectural consequences.

## High Findings

- High: Deployment hardening is missing in [UniPapers/settings.py](UniPapers/settings.py#L30), and Django’s own deploy check confirmed it. What it is: DEBUG is enabled whenever the app is not on Vercel, while SECURE_SSL_REDIRECT, SECURE_HSTS_SECONDS, SESSION_COOKIE_SECURE, and CSRF_COOKIE_SECURE are not set. Why dangerous: a misrouted deployment, staging leak, or proxy misconfiguration can expose debug behavior, weak cookies, and downgrade paths. Exploitability: moderate; attackers benefit from insecure transport or a misdeployment rather than from a direct code bug. Real-world impact: session theft over weaker channels, leaked stack traces, and weaker browser-side protections. Worst case: the app ships with a configuration that is functionally demo-grade instead of production-grade. Fix: set hard production defaults for secure cookies, HSTS, SSL redirect, and proxy SSL awareness; gate DEBUG on a positive development-only flag, not a negative deployment guess. Type: architectural and deployment-level.

- High: OAuth and session handling are too permissive in [UniPapers/settings.py](UniPapers/settings.py#L74), [UniPapers/settings.py](UniPapers/settings.py#L76), and [UniPapers/settings.py](UniPapers/settings.py#L86). What it is: account verification is disabled, logout can happen on GET, and OAuth tokens are stored. Why dangerous: this expands the attack surface for session abuse and increases the blast radius of a database compromise. Exploitability: moderate; token theft becomes much more valuable when tokens are stored and logout behavior is weak. Real-world impact: lingering tokens, forced logout abuse, and unnecessary secret retention in the database. Worst case: a database breach becomes a second-order compromise of identity providers or long-lived user sessions. Fix: stop storing OAuth tokens unless they are actively used, disable GET logout, and tighten the account/session policy to the minimum needed for Google-only login. Type: architectural.

- High: The preview endpoint disables clickjacking defenses in [home/views.py](home/views.py#L511) through [home/views.py](home/views.py#L531). What it is: the document preview is explicitly marked xframe_options_exempt and returned inline. Why dangerous: third-party sites can embed the preview page, and the app gives no browser-level framing protection for a route that serves user documents. Exploitability: moderate; this is especially relevant for owner-only or moderator-visible files. Real-world impact: clickjacking exposure and unnecessary embedding of document content on hostile pages. Worst case: a malicious site frames a sensitive document view and uses it as part of a social-engineering or privacy abuse chain. Fix: remove the exemption unless there is a documented requirement for framing, and if embedding is required, constrain it to known trusted origins. Type: implementation-level with security-policy implications.

- High: Third-party CDN dependencies are loaded without integrity protection in [home/templates/home/base.html](home/templates/home/base.html#L2) and [home/templates/home/base.html](home/templates/home/base.html#L635). What it is: Bootstrap and Google Fonts are fetched from external CDNs directly in the base template. Why dangerous: a CDN compromise, content injection, or restrictive network environment can alter or break the site’s UI layer. Exploitability: low to moderate, but the blast radius is broad because this affects every page. Real-world impact: supply-chain exposure and avoidable availability risk. Worst case: a malicious CDN response injects content into the public app shell. Fix: self-host critical assets or add Subresource Integrity and a clear asset pinning strategy. Type: architectural and supply-chain.

- High: Upload failure messages leak raw exception text to end users in [home/views.py](home/views.py#L644). What it is: the upload handler catches broad exceptions and returns str(exc) to the browser. Why dangerous: internal file paths, storage errors, and implementation details can be disclosed to attackers. Exploitability: easy; an attacker just needs to trigger a failure path. Real-world impact: information disclosure that can help targeted probing or further exploitation. Worst case: an error response reveals enough about the internal environment to accelerate follow-on attacks. Fix: log the exception server-side, return a generic user-facing error, and keep diagnostic details out of the UI. Type: implementation-level.

## Medium Findings

- Medium: Reporting is spam-prone and unbounded in [home/views.py](home/views.py#L460) through [home/views.py](home/views.py#L474). What it is: every report appends another line to record.msg with no deduplication or rate limit. Why dangerous: a logged-in user can spam reports, bloat storage, and drown admins in noise. Exploitability: easy for any authenticated account. Real-world impact: report spam, noisy moderation queues, and needless growth of the record row. Worst case: a malicious user weaponizes reporting as a storage and moderation-denial mechanism. Fix: rate-limit reports, deduplicate repeated reports per user and record, and store reports in a separate normalized table. Type: implementation-level and data-model.

- Medium: Search, profile, and preview traffic are heavier than they need to be in [home/views.py](home/views.py#L395), [home/views.py](home/views.py#L430), [home/views.py](home/views.py#L433), [home/views.py](home/views.py#L670), [home/views.py](home/views.py#L677), and [home/views.py](home/views.py#L678). What it is: each request computes multiple counts, distinct lists, and filtered aggregates, and the result cards fan out into document preview fetches. Why dangerous: under burst traffic, the app burns database and storage bandwidth on repeated lookups and full preview fetches. Exploitability: easy; simple browsing can generate a lot of back-end work. Real-world impact: slow search, slow profile pages, and unnecessary file serving load. Worst case: a spike of search traffic turns into a storage and database hotspot. Fix: cache filter metadata, precompute or denormalize frequent aggregates, and replace heavy preview iframes with cheap thumbnails or static previews. Type: architectural and performance.

- Medium: Search and profile pages load full document previews directly inside result cards in [home/templates/home/search.html](home/templates/home/search.html#L678) and [home/templates/home/profile.html](home/templates/home/profile.html#L817). What it is: every PDF result card embeds the preview endpoint in an iframe. Why dangerous: one page view can trigger many large file fetches. Exploitability: easy for any visitor. Real-world impact: bandwidth spikes, slow page rendering, and avoidable storage hot spots. Worst case: a few users repeatedly open search pages and generate enough file traffic to slow the site for everyone. Fix: generate small thumbnails or cached preview images asynchronously, and never load the full file just to render a list card. Type: implementation-level with scaling consequences.

- Medium: The application lacks rate limiting on the expensive public surfaces. The upload path is throttled in [home/views.py](home/views.py#L572), but search, report, view, and preview are not. Why dangerous: hostile users can hammer search and preview endpoints to consume CPU, database, and storage bandwidth. Exploitability: easy with automation. Real-world impact: application-level denial of service and degraded responsiveness for normal users. Worst case: burst traffic or a small botnet causes visible site slowdowns without ever touching infrastructure-level DDoS. Fix: rate-limit search, report, preview, and write operations separately, and add edge caching where possible. Type: architectural.

- Medium: The file validation path is only as strong as the optional python-magic dependency in [home/views.py](home/views.py#L205). What it is: if python-magic is missing, MIME detection falls back to extension-based guessing. Why dangerous: the code silently weakens content inspection when a nonessential dependency is absent. Exploitability: moderate in misconfigured environments. Real-world impact: weaker file-type validation and reduced confidence in upload safety. Worst case: the deployment runs with a false sense of protection around file contents. Fix: make binary MIME inspection a hard dependency or fail closed when it is unavailable. Type: implementation-level.

- Medium: The upload pipeline has no aggregate size cap and no cleanup strategy for partial failures in [home/views.py](home/views.py#L596) through [home/views.py](home/views.py#L644). What it is: the handler validates each file independently and saves files before the request is fully complete. Why dangerous: a user can send multiple large files in one request, and if the transaction fails after storage writes, orphaned files remain. Exploitability: easy through normal uploads. Real-world impact: wasted bandwidth, wasted storage, and stale pending files. Worst case: repeated failed uploads create a storage mess that is expensive to clean up manually. Fix: enforce a total request-size cap, use a staged upload record, and add cleanup for failed writes. Type: implementation-level and operational.

- Medium: No database indexes are visible on the hot filter columns in [home/models.py](home/models.py#L36) through [home/models.py](home/models.py#L60), while the views repeatedly filter on uploaded_email, status, year, program, semester, term, and title. Why dangerous: query cost will rise quickly as the dataset grows. Exploitability: not an attack by itself, but it makes every attack or traffic spike more expensive to absorb. Real-world impact: slow search and profile pages, especially under concurrent access. Worst case: a modest paper catalog becomes a high-latency site because the database is doing table scans for routine pages. Fix: add indexes for the fields used in filters and owner lookups, and verify with query plans before launch. Type: architectural and performance.

## Low Findings

- Low: The test suite is effectively empty in [home/tests.py](home/tests.py#L1). What it is: there are no behavioral tests around authentication, uploads, delete flows, reports, or preview permissions. Why dangerous: regressions will ship unnoticed, and the most sensitive code paths are not covered. Exploitability: indirect, but it increases the chance of shipping broken security logic. Real-world impact: slow remediation and higher production risk. Worst case: a later change breaks access control or file handling and nobody notices until after deployment. Fix: add focused tests for permissions, upload validation, approval behavior, and deletion. Type: process and quality.

- Low: Several code paths are dead, duplicated, or structurally confusing, including the unused login_page function and duplicated auth routes. Why dangerous: this makes security review harder and increases the chance that future changes touch the wrong entry point. Exploitability: low. Real-world impact: maintainability drag and higher regression risk. Worst case: a future auth change is made in one place but not the other and the live behavior diverges. Fix: remove dead code, consolidate auth routing, and document the actual login/logout path. Type: maintainability.

- Low: Ownership checks rely on email equality in [home/views.py](home/views.py#L724), [home/views.py](home/views.py#L670), and [home/views.py](home/views.py#L677). What it is: uploaded records are tied to the user’s current email address rather than a durable immutable owner key. Why dangerous: if identity data changes, ownership semantics can drift. Exploitability: low in the current Google-only flow, but it is a fragile design choice. Real-world impact: broken profile history or incorrect ownership decisions after account data changes. Worst case: a future auth migration silently breaks access checks. Fix: store and use a foreign key to the user model for ownership and uploader identity. Type: architectural.

## Potential Risks

- Potential Risk: The app relies on a conditional import for python-magic. If that package is missing in a deployment image, file-type validation silently weakens instead of failing closed.

- Potential Risk: The codebase does not show any queue, async task, or cache layer. Under burst traffic, all heavy work appears to happen inline in the request path.

- Potential Risk: I did not see backup, restore, monitoring, or alerting configuration in the repository. That is an operational gap, not just a code gap.

## Top 10 Most Dangerous Issues

1. Hardcoded SECRET_KEY in [UniPapers/settings.py](UniPapers/settings.py#L25).
2. Filesystem-based approval and move logic that conflicts with Cloudinary/Vercel in [home/models.py](home/models.py#L65) and [UniPapers/settings.py](UniPapers/settings.py#L137).
3. File overwrite risk on approval because approved paths are derived from raw basenames in [home/models.py](home/models.py#L88).
4. Missing production security hardening in [UniPapers/settings.py](UniPapers/settings.py#L30).
5. OAuth/session hardening too loose, especially token storage and GET logout in [UniPapers/settings.py](UniPapers/settings.py#L76) and [UniPapers/settings.py](UniPapers/settings.py#L86).
6. Clickjacking protection disabled on the preview endpoint in [home/views.py](home/views.py#L511).
7. Raw exception text is returned to users on upload failures in [home/views.py](home/views.py#L644).
8. Preview-heavy search/profile pages trigger repeated document fetches in [home/templates/home/search.html](home/templates/home/search.html#L678) and [home/templates/home/profile.html](home/templates/home/profile.html#L817).
9. Reporting can be spammed and will grow the record text field without bounds in [home/views.py](home/views.py#L460) through [home/views.py](home/views.py#L474).
10. The app has no meaningful automated test coverage in [home/tests.py](home/tests.py#L1).

## Immediate Fixes Before Deployment

1. Move SECRET_KEY, database credentials, and cloud credentials out of source control immediately and rotate anything that has already been exposed.
2. Set secure production defaults now: DEBUG off by default, secure cookies on, HSTS on, SSL redirect on, and proxy SSL handling configured correctly.
3. Replace the storage-move approval flow with a storage-agnostic design that does not rely on local filesystem renames.
4. Stop overwriting files by basename; generate unique final storage keys and verify collisions before finalization.
5. Remove raw exception leakage from upload handling and replace it with structured server-side logging.
6. Rate-limit search, preview, report, and write endpoints separately.
7. Replace PDF iframe fan-out with cheap thumbnails or cached preview images.
8. Add database indexes for the frequently filtered and ownership fields, then verify the query plans under realistic row counts.
9. Add tests for authentication, authorization, upload validation, delete permissions, and preview access control.
10. Decide whether the application is really cloud-storage based or filesystem based, then make the code and deployment match one architecture instead of both at once.

## Scalability Risks Under Traffic Spikes

- Search and profile pages perform repeated aggregate and distinct queries on every request, which will become a database hotspot as the catalog grows.
- PDF result cards load full preview endpoints, so ordinary browsing can fan out into many large file transfers.
- Uploads are synchronous and do not appear to use background jobs, so long uploads and moderation actions block request workers.
- There is no visible cache layer for filter metadata or result summaries.
- There is no visible queue architecture for expensive preview generation, moderation side effects, or storage cleanup.
- The app does not show any explicit connection pooling or edge-cache strategy in the repository.

## Production Launch Readiness Verdict

Verdict: not ready for public launch.

The application is close enough to demo that it may look finished, but it is not operationally mature enough for real users and definitely not mature enough for hostile internet traffic. The current state will not survive a serious security review or a real launch spike without exposing users to session risk, storage failures, and avoidable performance collapse. Fix the critical and high items first, then retest the deployment model from scratch before you let this go live.
